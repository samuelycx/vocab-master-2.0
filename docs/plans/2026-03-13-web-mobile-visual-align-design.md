# Web Mobile Visual Align Design

**目标**

在不改动核心业务逻辑的前提下，把网页版的手机端首页、学习页、设置页快速拉回到微信小程序当前版本的视觉风格，使用户在手机宽度下获得“同一产品”的连续体验。

**范围**

- 包含：
  - Web 首页
  - Web 学习页
  - Web 设置页
- 不包含：
  - PK 页视觉复刻
  - 词表、成就墙、排行榜、社交页的完整视觉还原
  - 桌面端精细适配

**设计原则**

1. 移动端优先
   - 以 390px 左右宽度为核心目标尺寸。
   - 桌面端先采用居中容器策略，不为桌面单独设计一套布局。

2. 复刻气质，不重写逻辑
   - 保留现有 Web 登录态、学习流程、状态管理、API 调用。
   - 主要调整布局层次、视觉 token、卡片比例、操作区节奏。

3. 第一轮先做 90% 还原
   - 优先恢复小程序的“看起来像”和“用起来像”。
   - 不追求字体渲染、像素级间距、浏览器滚动行为 100% 一致。

**方案比较**

## 方案 A：视觉复刻，结构微调

做法：
- 保留现有组件拆分和数据流。
- 小范围调整模板结构。
- 大部分工作落在布局、尺寸、色彩、阴影、圆角和组件层次上。

优点：
- 实施快。
- 风险低。
- 不容易把登录、学习、设置逻辑改坏。

缺点：
- 不是逐像素复刻。

推荐采用本方案。

## 方案 B：组件级重建

做法：
- 用小程序页面结构重新组织 Web 组件。

优点：
- 最接近原页面结构。

缺点：
- 变更范围大。
- 联调风险高。
- 容易和现有 Web 功能迁移工作冲突。

## 方案 C：纯样式覆盖

做法：
- 几乎不动模板，只通过 CSS 覆盖。

优点：
- 最快起手。

缺点：
- 很多位置和比例对不齐。
- 最终容易停留在“配色像，但不是同款页面”的状态。

**首页设计**

目标：
- 从当前偏 Web 的卡片仪表盘，调整为更像小程序首页的单列卡片节奏。

设计要点：
- 顶部 Hero 区回到“小程序主卡 + 今日进度”的视觉节奏。
- 强化主任务入口“开始学习 30 词挑战”。
- 次级入口采用两列卡片，但保持小程序式更厚的圆角和更浅的阴影。
- 成就入口恢复为小程序式模块感，信息密度更紧凑。
- 底部导航保持 Web 当前结构，但样式往小程序靠拢。

**学习页设计**

目标：
- 回到小程序式“顶部进度 + 居中单词卡 + 两列选项 + 底部提交感”的布局印象。

设计要点：
- 顶部使用更接近小程序的返回按钮、题号和生命/连击信息布局。
- 单词卡更高、更柔和，卡片背景和留白更接近小程序。
- 选项按钮改成小程序风格的圆角矩形卡。
- 降低 Web 风格的 hover 感，强调按压感和稳定对齐。
- 背景氛围改为更接近小程序的柔和纯色/渐变，不使用过强的 Web 装饰。

**设置页设计**

目标：
- 回到小程序当前的“分区卡片 + 用户卡 + 设置项 + 危险操作区”的视觉层次。

设计要点：
- 顶部标题和返回区更接近小程序。
- 用户卡改成更强识别度的大卡。
- 音效、主题等设置项统一成小程序式列表卡片。
- 保留现有功能，但降低后台感和桌面感。
- 危险操作区独立成明显但不刺眼的底部区块。

**视觉系统调整**

需要统一的基础 token：
- 背景色
- 主卡片底色
- 主按钮色
- 次级强调色
- 圆角层级
- 阴影层级
- 字重和字号层级
- 手机端页面最大宽度

设计策略：
- 先不推翻现有主题系统。
- 在现有 `academic/playful` 基础上，让默认手机端视觉优先靠近当前小程序版本。
- 这轮优先“页面形态一致”，不是先做多主题精修。

**技术实现策略**

- 尽量复用已有组件，不新增无必要抽象。
- 修改重点文件：
  - `client/src/components/Dashboard.vue`
  - `client/src/components/GameArena.vue`
  - `client/src/components/Settings.vue`
  - `client/src/App.vue`
  - `client/src/style.css`
- 通过测试保护现有登录和核心入口。
- UI 变更后至少执行：
  - 现有前端测试
  - 前端 build
  - 本地手机宽度手动预览

**验收标准**

1. 手机宽度下，首页、学习页、设置页视觉明显贴近小程序版本。
2. 登录后进入首页的主链路不受影响。
3. 开始学习、进入设置、退出登录、重置进度等关键行为不回归。
4. 前端测试和构建保持通过。

**风险与处理**

- 风险：现有组件有较强 Web 布局倾向，样式修改可能牵动交互。
  - 处理：先通过测试锁住登录、学习入口、设置动作，再改样式。

- 风险：桌面端显示可能不如手机端自然。
  - 处理：本轮明确以手机端优先，桌面仅做居中保底。

- 风险：小程序视觉还原过程中，可能暴露旧的文案或状态字段不一致。
  - 处理：仅做小范围兼容修正，不在本轮扩散到新功能开发。
