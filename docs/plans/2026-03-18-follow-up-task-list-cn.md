# 2026-03-18 后续任务清单（中文）

> [!note]
> 本清单来自 2026-03-18 这轮 `server + web + deploy` 修复之后明确延期的事项。
> 目标不是“记录想法”，而是把后续工作拆成可以单独立项、单独验收的任务。

## 执行原则

- 每个任务单独建分支，不与日常修复混提
- 数据治理类任务优先做只读审计，再做写入修复
- 小程序相关任务单独验收，不与当前 Web/Server 发布绑定
- 涉及云端凭据、线上数据清理时，先备份，再执行，再留痕

## 任务 1：小程序独立项目化与能力对齐

> [!todo]
> 目标：把 `miniprogram` 从“背景记录状态”推进到“可单独开发和验收”的项目状态。

### 范围
- 明确小程序当前云开发依赖、环境变量和部署方式
- 盘点与 `web` 的功能差异
- 确认哪些接口仍是 stub / 假成功实现

### 执行动作
- [ ] 盘点 `miniprogram` 当前可运行链路与云环境依赖
- [ ] 列出与 `web` 不一致的功能页面和接口能力
- [ ] 标记所有假成功返回或未接后端的 API
- [ ] 输出一份小程序独立设计/验收文档

### 验收标准
- 有一份可执行的 `miniprogram` 范围说明
- 有一份接口对齐差异表
- 明确哪些功能先做、哪些继续延期

## 任务 2：词库乱码审计后的数据修复

> [!todo]
> 目标：把“发现问题”推进到“可批量修复并可回归验证”的数据治理任务。

### 范围
- 仅处理词库文本质量问题
- 不改接口协议，不顺带做业务功能重构

### 执行动作
- [ ] 基于既有审计规则生成完整异常清单
- [ ] 区分“可脚本修复”和“需人工确认”的问题类型
- [ ] 先在备份数据库上做修复演练
- [ ] 输出修复前后 diff 与抽样验证结果
- [ ] 再决定是否写回正式数据源

### 验收标准
- 有异常清单和问题分类
- 有修复脚本或人工处理规范
- 有修复前后对比与回归记录

## 任务 3：排行榜虚拟账号清理

> [!warning]
> 目标：在不误伤真实用户的前提下清理虚拟账号和脏数据。

### 范围
- 仅处理排行榜相关异常账号/数据
- 不顺带修改积分算法或 UI 展示逻辑

### 执行动作
- [ ] 定义“虚拟账号”判定规则
- [ ] 导出待清理账号清单并人工复核
- [ ] 备份相关用户与排行榜数据
- [ ] 执行清理并重新计算排行榜展示结果
- [ ] 记录清理名单、时间和影响范围

### 验收标准
- 有可追溯的判定规则
- 有清理前备份
- 清理后排行榜结果可复核

## 任务 4：TTS 方案升级与凭据治理

> [!todo]
> 目标：提升播报自然度，并把本地/云端凭据管理纳入可控范围。

### 范围
- 评估更自然的 TTS 服务或参数方案
- 处理本地生成脚本、音频产物和云端凭据治理

### 执行动作
- [ ] 评估 2 到 3 种更拟人化的 TTS 方案
- [ ] 对比成本、音色、延迟、批量生成能力
- [ ] 轮换当前云端服务账号密钥
- [ ] 把凭据迁出仓库目录，改为环境变量或安全存储
- [ ] 明确 `tts/` 目录后续是工具目录、产物目录还是独立项目

### 验收标准
- 有方案对比结论
- 有新凭据存放方案
- 有批量生成或回退预案

## 任务 5：头像上传安全增强与资料保存原子化

> [!danger]
> 目标：补齐资料系统剩余的安全与一致性缺口。

### 范围
- 头像上传 MIME/大小/扩展名限制
- 失败处理、存储收口、资料保存一致性

### 执行动作
- [ ] 为头像上传增加 MIME 白名单与大小限制
- [ ] 拒绝非图片文件与异常扩展名
- [ ] 评估是否统一转码为安全图片格式
- [ ] 设计单接口资料更新或事务式保存流程
- [ ] 为部分成功/失败回滚补测试

### 验收标准
- 上传链路有明确白名单和限制
- 资料保存不会出现“部分成功但前端失败”的不一致
- 有覆盖失败分支的回归测试

## 推荐推进顺序

1. 先做 `任务 4`（TTS 凭据治理）和 `任务 5`（资料安全增强），因为都带安全/风险收口价值。
2. 再做 `任务 2`（词库数据修复）和 `任务 3`（排行榜清理），因为它们属于数据治理，适合单独演练。
3. 最后推进 `任务 1`（小程序独立项目化），避免再次和 Web/Server 发布耦合。
