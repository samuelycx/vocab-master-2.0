# Miniprogram Phase 1 Cloud Checklist

## 目的

用于小程序专项一期发版前后的手工验收，覆盖云函数、云数据库 / 云存储，以及微信开发者工具配置。

## 云函数检查

- 确认当前环境 ID 与发布目标一致，开发 / 体验 / 生产环境没有串用。
- 在微信开发者工具中执行“上传并部署：云端安装依赖”，确保本期涉及的云函数都完成重新部署。
- 抽查云函数日志，确认最近一次部署后不存在启动失败、依赖缺失或权限报错。
- 手工验证登录、用户信息拉取、排行榜或本期涉及的关键云函数调用至少各成功一次。
- 若云函数依赖环境变量或扩展配置，确认开发者工具与云托管后台中的值一致。

## 云数据库检查

- 确认 `pk_rooms` 集合存在，读写权限符合当前一期方案。
- 手工发起一次 PK 匹配，确认会创建 `waiting` 房间、第二个玩家加入后切换为 `playing`。
- 在对战过程中观察 `players[].score` 是否持续更新，结束后是否写入 `status=ended`、`winnerId`、`endedAt`、`endReason`。
- 等待超时或主动退出后，确认房间会被删除或标记结束，没有长期残留无效数据。
- 抽查是否存在大量历史 `waiting` / `ended` 房间未清理，必要时执行人工清理或观察自动清理逻辑。

## 云存储检查

- 确认默认云存储空间可访问，没有达到容量或带宽告警阈值。
- 抽查用户头像、词库相关静态资源或本期依赖资源是否能正常下载与展示。
- 如使用临时文件 URL，确认未出现过期失效导致的资源加载失败。
- 若体验版 / 生产版使用不同资源目录，确认资源已同步到目标环境。

## 微信开发者工具检查

- 使用本期目标版本的微信开发者工具打开项目，并确认已勾选使用正确的云开发环境。
- 清理编译缓存后重新构建一次，确认没有因为旧依赖缓存导致的误报。
- 检查合法域名、 request / upload / download 白名单与当前小程序后台配置一致。
- 在真机调试或体验版中完成一次登录、开始 PK、完成 PK、退出 PK 的完整流程。
- 关注控制台 `watch`、云调用和存储资源加载日志，确认不存在持续性报错。

## Git 版本证据检查

- 发布前先在 `miniprogram/` 目录执行 `npm run release:verify`。
- 发布前生成 `release-manifest.json`，并确认其中的 `commit` 来自目标 `origin/main` SHA。
- 记录本次发布 `scope`（`miniprogram` / `cloudfunctions` / `full`）以及 `cloudFunctions` 列表。
- 发布后把 `release-manifest.json` 中的 `commit` 和验收结论一并登记到发版记录中。

## 发布前签收建议

- 至少由 1 名开发和 1 名测试分别在开发者工具与真机上完成上述检查。
- 对发现的问题记录“环境 / 复现步骤 / 日志截图 / 是否阻塞发版”。
- 若发现需修改共享文件（例如接口层或页面层），不要在一期低风险清理中直接处理，转交主线程统一收敛。
