# 2026-03-19 小程序微信身份边界设计（中文）

> [!abstract]
> 本设计用于明确一个新的长期边界：`miniprogram` 的身份体系以微信云开发上下文为主，而 `web/server` 不承担微信身份登录能力。
> 从现在开始，“微信账号登录”不是等待 Web 端同步的缺口，而是小程序独立产品面的长期分化项。

## 一、问题背景

- 小程序当前运行在微信云开发语境下，实际身份来源是 `cloud.getWXContext().OPENID`
- `web/server` 当前不需要微信登录，也没有必要为了“小程序身份入口”反向改造成同一套体系
- 如果不把这件事定义为正式边界，后续很容易再次把它误判成：
  - “Web 还没补齐”
  - “小程序没和服务器同步”
  - “同一功能为什么两端不一样”

> [!warning]
> 真正需要避免的，不是“身份入口不同”，而是“核心业务语义漂移”。

## 二、设计结论

### 结论一句话

`miniprogram` 与 `web/server` 允许在 **身份体系和登录方式** 上长期分化，但必须在 **核心业务语义** 上持续对齐。

### 允许长期分化的部分

- 登录入口
- 身份来源
- 用户主键查询方式
- 端侧特有权限与平台能力

在当前项目里，具体表现为：

- 小程序：
  - 主身份来自微信上下文
  - 以 `openid` 作为核心查询键
  - 登录属于云函数 / 云开发职责
- Web / 本地服务器：
  - 不要求支持微信登录
  - 保持当前自有身份体系
  - 不把“小程序微信身份缺失”当成 Web 待补功能

### 不允许随意分化的部分

- canonical user snapshot 字段语义
- 资料字段语义
- 排行榜展示字段语义
- 学习进度、等级、XP、答对数等核心业务含义

> [!success]
> 也就是说：**身份层分化，业务层对齐**。

## 三、为什么这是推荐方案

### 方案 A：强行统一成一套登录体系

**不推荐**

- 会把微信云开发身份体系和 `web/server` 身份体系硬绑在一起
- 需要额外处理跨端鉴权、绑定、回填和迁移问题
- 当前收益明显低于改造成本

### 方案 B：身份层长期分化，业务层持续对齐

**推荐**

- 尊重当前两端真实运行时差异
- 不会把“平台身份入口”误当成“同步失败”
- 仍然保留共享业务语义，避免排行榜、资料、进度再次漂移

### 方案 C：小程序彻底脱离当前仓库治理

**暂不采用**

- 边界会更清楚，但现在拆分过早
- 会打断当前已经建立好的 release manifest、runbook 和同步治理体系

## 四、架构边界定义

### 4.1 小程序身份边界

- 小程序主身份为微信身份
- 当前事实上的登录动作是：
  - 小程序客户端调用云函数
  - 云函数从微信上下文读取 `OPENID`
  - 以 `openid` 查询 / 创建 / 更新用户
- 这条链路是否完整、是否可发布，属于 **小程序专项验收项**

### 4.2 Web / Server 身份边界

- `web/server` 不负责复刻微信身份登录
- `web/server` 的验收清单中不再包含“小程序微信身份能力”
- 后续如果要支持跨端账号绑定，应当作为新需求单独设计，而不是默认并入当前登录模型

### 4.3 同步治理边界

- 以后“是否同步”主要回答：
  - Git SHA 是否一致
  - release manifest 是否一致
  - 发布目标是否已经按自己的边界验收
- 不再用“功能完全一样”来定义“是否同步成功”

## 五、验收标准调整

### 小程序需要额外验收

- 微信身份登录是否真正可用
- 是否能从微信上下文正确拿到 `openid`
- 首次登录是否创建用户
- 再次登录是否恢复同一用户
- 设备绑定 / 封禁等微信身份相关逻辑是否生效

### Web / Server 不再验收

- 微信身份登录
- `openid` 驱动的登录入口
- 小程序端特有的微信授权流程

更新后的统一口径应该是：

- “小程序与 Web 不要求登录方式一致”
- “微信身份属于小程序长期分化项”
- “跨端需要保持一致的是业务语义，不是平台身份入口”

## 七、非目标

- 本设计不要求现在立刻重做微信登录实现
- 本设计不要求 Web 新增微信身份能力
- 本设计不直接处理 PK 鉴权二期重构
- 本设计不把 `openid` 与 `userId` 的跨端绑定关系扩展成新账号系统

## 八、下一步建议

> [!tip]
> 下一步不是继续争论“两端为什么不一样”，而是把边界写进文档和验收：
>
> 1. 更新小程序专项设计与 follow-up 文档
> 2. 更新 release runbook / checklist，把微信身份列为小程序独立验收项
> 3. 后续若要真正补“微信登录闭环”，单独作为小程序功能任务推进
